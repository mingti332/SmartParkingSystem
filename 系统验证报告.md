## 一、系统架构概述

JavaFX 桌面应用 + JDBC 直连 MySQL，三角色体系：

| 角色 | 编码 | 功能 |
|------|------|------|
| 管理员 | ADMIN | 用户管理、停车场管理、车位管理、计费规则、结算审核、报表、操作日志 |
| 车位所有者 | OWNER | 我的车位、名下车位预约查看、收益明细 |
| 车主 | CAR_OWNER | 查询可用车位、提交预约、取消预约、入场登记、出场支付、停车/支付记录 |

## 二、代码错误与缺陷汇总

### [严重] 1. 预约状态机不完整 —— 入场后预约状态未流转

**位置**：`ParkingRecordServiceImpl.entry()` (行35-90)

**问题**：通过预约入场时，代码仅校验了预约状态（PENDING/ACTIVE），但入场成功后**从未将预约状态更新为 ACTIVE**；出场支付时（`exitAndPay` 行 152-201）也**从未将预约标记为 COMPLETED**。预约将永远停留在 PENDING 状态，导致：

- 该预约在 `hasConflict()` 中永久占位，阻止同车位后续相同时段的预约
- 数据库中 Reservations 表的 status 字段失去实际意义
- 车主无法再次预约同一车位的重叠时段，即使该预约早已完成

**修复建议**：在 `entry()` 成功插入 ParkingRecord 后，增加 `reservationDao.updateStatus(conn, reservationId, "ACTIVE")`；在 `exitAndPay()` 中增加将关联预约状态更新为 "COMPLETED" 的逻辑。

---

### [严重] 2. UI 层入场预约匹配使用了错误的状态值

**位置**：`DashboardFactory.java` 行 783-796 (`parkingTab` -> entry button)

**问题**：代码检查 `"RESERVED".equals(r.getStatus())`，但实际上预约创建后预约状态是 **PENDING**（车位状态才是 RESERVED）。这导致**预约入场分支永远不会被命中**，预约用户会直接走"空闲车位入场"分支，绕过了预约验证。

```java
// 错误：检查的是预约记录的状态，但预约记录状态是 PENDING，不是 RESERVED
if ("RESERVED".equals(r.getStatus())  // ← BUG: 应为 "PENDING" 或 "ACTIVE"
```

**影响**：预约车主入场时，系统不会使用预约 ID 关联入场记录，`reservation_id` 字段为 NULL。

---

### [中等] 3. 修改密码未校验旧密码

**位置**：`AuthServiceImpl.changePassword()` (行148-162)

**问题**：方法签名接收了 `oldPassword` 参数，但方法体内**完全没有使用该参数进行校验**。任何人只要知道用户 ID 就可以修改密码。

```java
public void changePassword(Long userId, String oldPassword, String newPassword) {
    // oldPassword 未被使用！直接更新新密码
    if (userDao.updatePassword(userId, encodedNewPassword) == 0) { ... }
}
```

---

### [中等] 4. 预约取消仅限 PENDING 状态，ACTIVE 状态预约无法取消

**位置**：`ReservationDaoImpl.cancelById()` (行75-86)

**问题**：SQL 条件是 `WHERE reservation_id = ? AND user_id = ? AND status = 'PENDING'`。一旦预约变为 ACTIVE（如果状态机被修复），则无法取消。车主已入场但希望提前结束的预约无法处理。

---

### [中等] 5. 服务层缺少停车场营业时间校验

**位置**：`ReservationServiceImpl.reserve()` / `reserveByLotAndType()`

**问题**：营业时间校验**仅存在于 UI 层**（`DashboardFactory.java` 行514-537），服务层的 `reserve()` 和 `reserveByLotAndType()` 方法完全没有对停车场 `openTime`/`closeTime` 的校验。如果有人通过 API 直接调用（或未来改造为 Web 接口），可以预约到营业时间之外的时段。

**同样的问题**存在于 `ParkingRecordServiceImpl.entry()` / `entryByLotAndType()`——入场也没有校验停车场营业时间。

---

### [中等] 6. 入场后共享时间检查逻辑有缺陷

**位置**：`ParkingRecordServiceImpl.entry()` 行70-76

**问题**：非预约入场时检查 `shareEndTime`：
```java
if (space.getShareEndTime() != null && !now.toLocalTime().isBefore(space.getShareEndTime())) {
    throw new ServiceException("当前已超过该车位共享结束时间");
}
```
使用 `!isBefore` 意味着**等于**结束时间的时刻也会被拒绝，而实际上在结束时间那一分钟应该仍然允许入场。应使用 `isAfter`。

---

### [轻微] 7. 车位 ID 人工管理存在并发风险

**位置**：`ParkingSpaceDaoImpl.insert()` (行20-47)、`PricingRuleDaoImpl.insert()` (行37-57)

**问题**：使用 `findReusableId()` 查找最小可用 ID，失败则重试最多 5 次。当有并发插入时，两个线程可能拿到相同的 ID，其中一个会因为主键冲突而失败。虽然生产环境通常使用 MySQL 的 AUTO_INCREMENT，但这种手动管理方式在并发场景下不够健壮。

---

### [轻微] 8. 出场支付未校验停车场营业时间

**位置**：`ParkingRecordServiceImpl.exitAndPay()` (行152-201)

**问题**：出场时没有检查停车场是否在营业时间内。如果用户在停车场关闭后出场，系统仍然会计算费用并完成支付。

---

### [轻微] 9. 注册接口与用户管理接口手机号校验不一致

**位置**：`AuthServiceImpl.register()` vs `UserAdminServiceImpl.createUser()`

**问题**：前端注册时手机号是**必填项**（11 位数字），但管理员后台创建用户时手机号变为**可选**。这种不一致可能导致管理员创建的用户在某些功能中缺少必要的联系方式。

---

### [轻微] 10. 停车场 is24Hour() 方法判断条件过于严格

**位置**：`ParkingLot.java` 行70-75

**问题**：判断 24 小时营业需要 `openTime == MIDNIGHT && closeTime == MIDNIGHT`。但实际业务中，"00:00-24:00" 或 "00:00-23:59" 都可能表示全天营业，当前逻辑只能识别双 MIDNIGHT 的情况。

## 四、整体评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | ⭐⭐⭐⭐ | 清晰的三层架构(Entity-DAO-Service-UI)，角色分离明确 |
| 业务完整性 | ⭐⭐⭐ | 预约状态机不完整，缺少 ACTIVE/COMPLETED 状态流转 |
| 数据一致性 | ⭐⭐⭐⭐ | 事务管理到位（conn.commit/rollback），使用 Connection 传递 |
| 安全性 | ⭐⭐⭐ | 密码加密存储、登录冷却机制好，但改密未验证旧密码 |
| 输入校验 | ⭐⭐⭐ | UI 层校验较全，服务层缺少营业时间等关键校验 |
| 用户体验 | ⭐⭐⭐⭐ | JavaFX UI 设计完整，支持内联编辑、筛选、分页 |

## 五、优先修复建议

1. **修复预约状态机** — 在 `entry()` 中将预约置为 ACTIVE，在 `exitAndPay()` 中将预约置为 COMPLETED
2. **修复 UI 入场预约匹配** — `DashboardFactory.java:786`，`"RESERVED"` 改为 `"PENDING"`
3. **修复改密未校验旧密码** — 在 `changePassword()` 中增加旧密码比对
4. **服务层补充营业时间校验** — 在 `ReservationServiceImpl` 和 `ParkingRecordServiceImpl` 中加入停车场营业时间验证
5. **修复 shareEndTime 边界判断** — 将 `!isBefore` 改为 `isAfter`
